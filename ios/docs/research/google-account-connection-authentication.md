# Research: Native iOS Google Account Connection authentication

_Researched: 2026-07-17._

## Summary

Use **Google Sign-In for iOS** with an **iOS OAuth client ID** and request `https://www.googleapis.com/auth/calendar.readonly`; restore the one locally selected `GIDGoogleUser` on launch and treat the connection as valid only when that scope remains granted. This is Google’s supported, highest-level native path and delegates authorization UI, callback handling, token refresh, and persisted sign-in state to Google’s SDK; a hand-built `ASWebAuthenticationSession` authorization-code + PKCE flow is supported but creates substantially more security and lifecycle work. [Google iOS integration](https://developers.google.com/identity/sign-in/ios/start-integrating) [Google native-app OAuth](https://developers.google.com/identity/protocols/oauth2/native-app)

Reuse of Planner’s web backend is viable only as a **new native-client/backend protocol**, not by reusing the current GIS popup and encrypted `HttpOnly` cookie design unchanged. If Calendar calls will eventually be server-side, the iOS SDK can obtain a one-time server authorization code for a backend configured with a separate Web client ID; the backend may exchange and retain the refresh token, while iOS retains only an app session credential. [Google backend authentication](https://developers.google.com/identity/sign-in/ios/backend-auth)

## Findings

1. **Recommended option: Google Sign-In for iOS.** Google’s native integration directs iOS apps to configure an iOS OAuth client and use `GIDSignIn.sharedInstance.signIn(withPresenting:)`; the SDK exposes scope addition, previous-sign-in restoration, token refresh, sign-out, and disconnect. This is preferable to reproducing those responsibilities with raw OAuth endpoints. It is an official Swift Package (`https://github.com/google/GoogleSignIn-iOS`), so adopting it changes the app’s present “no third-party packages” state, but does not require Firebase. [Integration guide](https://developers.google.com/identity/sign-in/ios/start-integrating) [GIDSignIn reference](https://developers.google.com/identity/sign-in/ios/reference/Classes/GIDSignIn)

2. **Use an iOS OAuth client, not Planner’s Web client.** Create an OAuth 2.0 client of application type **iOS**, bound to Planner’s bundle identifier; configure `GIDClientID` and register the client ID’s reversed form as a URL scheme. The iOS app’s `onOpenURL`/application URL callback passes the URL to `GIDSignIn.handle(_:)`. A Web client ID is additionally needed only for backend authorization-code exchange (`GIDServerClientID`/`serverClientID`). [Create/configure iOS client](https://developers.google.com/identity/sign-in/ios/start-integrating#add_your_oauth_client_id_to_your_apps_configuration) [URL handling](https://developers.google.com/identity/sign-in/ios/sign-in#handle_the_authentication_redirect_url) [Backend auth](https://developers.google.com/identity/sign-in/ios/backend-auth)

3. **No client secret belongs in iOS.** Google classifies iOS as an installed application and says installed apps cannot keep secrets; the native client configuration therefore must not depend on a client secret embedded in the bundle. In the backend-code variant, only the server’s Web-client secret is used, and it remains on the server. [OAuth for installed apps](https://developers.google.com/identity/protocols/oauth2/native-app) [Backend code exchange](https://developers.google.com/identity/sign-in/ios/backend-auth)

4. **Scope choice: `calendar.readonly` is the direct fit.** Google defines `https://www.googleapis.com/auth/calendar.readonly` as permission to “see and download any calendar you can access using your Google Calendar”; it is read-only and covers the later Source Calendar and Calendar Event reads without write access. A least-privilege alternative for later implementation is the pair `calendar.calendarlist.readonly` plus `calendar.events.readonly`, but it is more complex and should be confirmed against every future endpoint. Calendar scopes are sensitive, so the OAuth consent screen, production status, user-data disclosures, and—when Google requires it—OAuth app verification must be completed before broad distribution. [Calendar scopes](https://developers.google.com/workspace/calendar/api/auth) [OAuth app verification](https://support.google.com/cloud/answer/13461325) [Google API Services User Data Policy](https://developers.google.com/terms/api-services-user-data-policy)

5. **Requesting the Calendar scope is authorization, even when this slice fetches nothing.** Google recommends requesting scopes in context and supports incremental authorization on iOS through `addScopes(_:presenting:)`. Planner can first establish Google identity and then request `calendar.readonly` as part of the same user-visible Connect action; alternatively it can defer Calendar consent until the feature first needs it. Because Google supports granular permission decisions, the app must inspect `grantedScopes`/`hasGrantedScopes` and must not show a connected Google Account Connection unless the required Calendar scope was actually granted. [Incremental authorization](https://developers.google.com/identity/sign-in/ios/additional-scopes) [OAuth best practices](https://developers.google.com/identity/protocols/oauth2/resources/best-practices)

6. **Durability with the SDK: restore, validate, refresh.** On startup call `restorePreviousSignIn`; Google documents this as restoring a previous session, and `GIDGoogleUser.refreshTokensIfNeeded` refreshes expired access/ID tokens before use. The SDK persists authorization state in the Keychain; Planner should retain only the single restored/current user and clear it on Disconnect on This Device. This supports app and device restarts, but cannot promise permanence after Google invalidates authorization, the user revokes access, token policy changes, or local secure state is removed. Do not promise survival after app uninstall. [Restore previous sign-in](https://developers.google.com/identity/sign-in/ios/sign-in#restore_the_previous_sign-in) [GIDGoogleUser reference](https://developers.google.com/identity/sign-in/ios/reference/Classes/GIDGoogleUser) [SDK migration notes on Keychain state](https://developers.google.com/identity/sign-in/ios/quick-migration-guide)

7. **Keychain is the storage boundary for native secrets.** If implementing direct OAuth, persist refresh tokens (and any durable backend session credential) in Keychain, not `UserDefaults`, SwiftData, files, logs, or source. Apple describes Keychain as storage for passwords, keys, and other small secret data; access should be restricted to this app unless intentional sharing is required. With Google Sign-In, let the SDK own Google token persistence rather than duplicating tokens into app storage. [Apple Keychain Services](https://developer.apple.com/documentation/security/keychain-services) [Google OAuth security best practices](https://developers.google.com/identity/protocols/oauth2/resources/best-practices)

8. **Disconnect and revocation are different operations.** `signOut()` removes local sign-in state; the SDK's `disconnect()` revokes the authorization and then signs the user out. Google documents that revocation removes all scopes granted to the project and invalidates tokens for every client registered under that project. Planner's **Disconnect on This Device** is deliberately local, so iOS must use `signOut()` and preserve the Google Authorization Grant and other Planner connections; global revocation is not part of this slice. [iOS disconnect/sign-out](https://developers.google.com/identity/sign-in/ios/disconnect) [OAuth token revocation](https://developers.google.com/identity/protocols/oauth2/web-server#tokenrevoke)

9. **Direct `ASWebAuthenticationSession` + authorization code + PKCE is supported, but second choice.** Google supports installed-app authorization through the system browser, an iOS client ID, a redirect URI/custom scheme, an authorization code, and PKCE (`code_challenge`/`code_verifier`); Apple’s `ASWebAuthenticationSession` provides the system authentication presentation and returns the callback URL. Planner would itself need to generate/store state and PKCE values, validate state and callback, exchange the code, securely persist/rotate the refresh token, refresh access tokens, handle partial grants and errors, and revoke/erase on disconnect. Google explicitly recommends using its platform libraries where available because correct OAuth implementation is security-sensitive. [Google native-app flow](https://developers.google.com/identity/protocols/oauth2/native-app) [ASWebAuthenticationSession](https://developer.apple.com/documentation/authenticationservices/aswebauthenticationsession) [OAuth native-app security, RFC 8252](https://www.rfc-editor.org/rfc/rfc8252)

10. **Direct-flow refresh behavior requires `access_type=offline`.** Google returns a refresh token for installed-app authorization when offline access is requested; later refreshes go to Google’s token endpoint. Refresh tokens can stop working after user revocation, six months of non-use, password/security events, token-count limits, or—critically for development—seven days when an external consent screen remains in Testing and requests non-basic scopes. The app must treat refresh failure/`invalid_grant` as an invalidated connection and require reconnection, never as a crash loop. [Google OAuth refresh-token rules](https://developers.google.com/identity/protocols/oauth2#expiration) [Installed-app token exchange](https://developers.google.com/identity/protocols/oauth2/native-app#exchange-authorization-code)

11. **The current web backend is reusable as infrastructure, not as-is as the native authentication boundary.** The existing Web client/GIS popup is a browser flow, while Google requires an iOS client for the native app. An `HttpOnly` cookie is intentionally inaccessible to app script, and `ASWebAuthenticationSession`’s browser cookie context is not a durable native API credential contract. A backend variant therefore needs: (a) iOS Google Sign-In configured with both iOS and server Web client IDs; (b) a one-time server auth code sent over TLS; (c) backend code exchange and existing encrypted refresh-token retention adapted to a user/device record; and (d) an app-session credential stored in Keychain, with authenticated native API endpoints and disconnect/revocation. This is an architectural inference from Google’s documented backend-auth flow and Apple’s cookie/storage boundaries. [Google backend auth](https://developers.google.com/identity/sign-in/ios/backend-auth) [Apple cookie storage](https://developer.apple.com/documentation/foundation/httpcookiestorage) [Apple Keychain](https://developer.apple.com/documentation/security/keychain-services)

12. **Backend trade-off.** Backend ownership keeps Google refresh tokens out of the app and can reuse encryption/revocation logic, but adds account/session mapping, CSRF/replay-safe code exchange, native API authentication, availability, and deletion obligations before this authorization-only slice receives value. Native SDK ownership is the smallest coherent implementation now; move Calendar calls and refresh-token custody server-side only when Planner has a concrete server-side data requirement. Google requires servers to validate the authorization result and use the one-time code rather than trusting client-provided identity fields. [Backend authentication](https://developers.google.com/identity/sign-in/ios/backend-auth) [OAuth server-side policies](https://developers.google.com/identity/protocols/oauth2/policies)

13. **Privacy manifest and App Store review must be planned, not guessed.** Apple lists `GoogleSignIn` among commonly used third-party SDKs that must include a privacy manifest and, for listed binary SDKs, a signature; Google Sign-In iOS release notes document privacy-manifest support. Pin a current SDK release and verify that its `PrivacyInfo.xcprivacy` is embedded in the archive; Planner still remains responsible for accurate App Privacy answers covering its own and dependencies’ collection/use of account and Calendar data. [Apple third-party SDK requirements](https://developer.apple.com/support/third-party-SDK-requirements/) [Google Sign-In iOS release notes](https://developers.google.com/identity/sign-in/ios/release) [Apple App Privacy details](https://developer.apple.com/app-store/app-privacy-details/)

14. **Sign in with Apple is likely not triggered, but review the final UX.** App Review Guideline 4.8 requires an equivalent privacy-preserving login when a third-party login authenticates the user’s primary app account, but includes an exception for apps that are clients of a specific third-party service and require direct sign-in to that service. Planner should present this as a **Google Account Connection for Google Calendar**, not generic Planner account authentication; if product scope later turns it into the app’s primary account system, Sign in with Apple must be reassessed. [App Review Guidelines 4.8](https://developer.apple.com/app-store/review/guidelines/#sign-in-with-apple)

15. **Use Google's supplied disconnected button.** Google's branding guidelines require compliant text, color, padding, and the standard multicolor “G” for app verification. Google Sign-In 9.2.0 supplies SwiftUI icon and wide button styles with localized Google-owned copy, so Planner can use the icon form in constrained headers and the wide form when space permits instead of maintaining a custom branded control. The connected profile capsule remains Planner-owned UI. [Google branding guidelines](https://developers.google.com/identity/branding-guidelines) [Google Sign-In SwiftUI button source](https://github.com/google/GoogleSignIn-iOS/blob/9.2.0/GoogleSignInSwift/Sources/GoogleSignInButton.swift)

16. **Google Sign-In 9.2.0 handles fresh reinstall cleanup.** Release 9.2.0 removes stale SDK Keychain entries when it detects a fresh app installation, matching Planner's requirement that reinstall starts disconnected. That behavior does not by itself establish a no-migration guarantee for a backup restored to another physical device, so Planner still needs its own non-migrating installation boundary if that product guarantee is enforced. [Google Sign-In 9.2.0 release](https://github.com/google/GoogleSignIn-iOS/releases/tag/9.2.0) [Fresh-install change](https://github.com/google/GoogleSignIn-iOS/pull/567)

## Decision matrix

| Option | Google support | Durable credential owner | Secret in iOS? | Fit for this slice |
|---|---|---|---|---|
| Google Sign-In iOS SDK, Calendar access native | First-party recommended integration; SDK handles restore/refresh and callback. [Docs](https://developers.google.com/identity/sign-in/ios/start-integrating) | SDK/Keychain | No client secret | **Recommended**: least custom security code; one new official package. |
| Direct OAuth code + PKCE in `ASWebAuthenticationSession` | Supported installed-app flow. [Docs](https://developers.google.com/identity/protocols/oauth2/native-app) | Planner/Keychain | No client secret | Valid only if avoiding the SDK is a hard constraint; materially larger auth surface. |
| iOS SDK + adapted web backend | Supported server-auth-code pattern. [Docs](https://developers.google.com/identity/sign-in/ios/backend-auth) | Backend stores Google refresh token; iOS stores app session | Web secret server-only | Best only if backend must perform Calendar work; current cookie/GIS flow is not directly reusable. |

## Proposed implementation boundary

- Add Google Sign-In 9.2.0 or a reviewed compatible 9.x release, including its supplied SwiftUI button; configure a new iOS OAuth client, reversed-client-ID URL scheme, and SwiftUI `onOpenURL` forwarding. [Google integration](https://developers.google.com/identity/sign-in/ios/start-integrating) [9.2.0 release](https://github.com/google/GoogleSignIn-iOS/releases/tag/9.2.0)
- On Connect, sign in and obtain `calendar.readonly` through incremental authorization; confirm the granted scope before publishing connected state. Keep exactly one current `GIDGoogleUser`. [Additional scopes](https://developers.google.com/identity/sign-in/ios/additional-scopes)
- On launch, call `restorePreviousSignIn`, then refresh if needed; on unrecoverable authorization failure, clear connected state and require Connect. [Restore](https://developers.google.com/identity/sign-in/ios/sign-in#restore_the_previous_sign-in)
- On Disconnect on This Device, call local `signOut()`, then clear Planner state without revoking the Google Authorization Grant. Do not fetch Source Calendars or Calendar Events in this slice. [Disconnect versus sign-out](https://developers.google.com/identity/sign-in/ios/disconnect)
- Before release, configure the consent screen/production status, privacy policy and verified domains as applicable; verify the archived SDK privacy manifest and complete App Privacy disclosures. [OAuth production readiness](https://developers.google.com/identity/protocols/oauth2/production-readiness/policy-compliance) [Apple privacy](https://developer.apple.com/app-store/app-privacy-details/)

## Gaps

- Google’s public docs do not promise a particular Keychain accessibility class or survival across uninstall/reinstall; verify actual SDK behavior on iOS 17+ and do not make either a product guarantee.
- Final OAuth verification requirements depend on consent-screen audience, publication state, branding, and Google’s review determination. Test with a production-mode project before App Store submission.
- Apple Review decisions are contextual. The Guideline 4.8 exception appears applicable only while Google is a service connection rather than Planner’s primary account login.
